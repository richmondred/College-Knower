Executive summary
Your core idea is strong, but the dataset and rules need a few important 2026 corrections before you
hand this to Codex. The biggest factual fix is that Louisiana Tech should not be in Conference USA for
the corrected 2026 quiz dataset: Louisiana Tech’s own athletics site says its 2026 football schedule
includes eight Sun Belt games and explicitly frames the season as a return to the Sun Belt
Conference. The Mountain West’s 2026 official standings already include UTEP , North Dakota State
and Northern Illinois, and the MAC’s 2026 official standings include Sacramento State and 
Massachusetts. Texas State’s move to the rebuilt Pac-12 was officially completed in mid-2025 and
recent reliable reporting describes it as the Pac-12’s eighth football-playing member for the 2026
launch, while the Pac-12 site’s member listing reflects the new league structure. In other words: your
prompt should be written against a versioned 2026 dataset, not against a hard-coded list typed from
memory. 
On implementation, the biggest quality gain will come from treating answer matching as a 
deterministic quiz engine, not a simple text box. Use a curated alias map, Unicode normalisation,
allowlist-style matching, token-boundary checks, and an explicit collision strategy for schools such as 
Texas / Texas A&M / Texas State / Texas Tech / North Texas and Miami / Miami (FL) / Miami (OH).
The feature you originally described as “duplicates” is better described as overlapping or cascading
auto-submit: in easy mode, one input may intentionally satisfy multiple answers if your rules allow it; in
medium and hard, it should not. For robust UX, ignore matches while IME composition is in progress,
persist quiz state with a versioned save object, recompute remaining time from an absolute deadline
rather than trusting interval ticks, and use visible focus, live-region announcements, and reduced-
motion variants by default. 
For leaderboard architecture, the simplest serious choice is usually Supabase + Postgres + serverless
validation. It gives you SQL structure, sane pricing for a first release, and built-in real-time options,
while still letting you keep the anti-cheat logic server-side. Firebase is still the fastest way to ship if you
want a highly managed client-friendly backend; Hasura is best if you want a GraphQL-first architecture
and do not mind the extra platform complexity. A client-only leaderboard is acceptable only for
prototype/demo use because time, score and difficulty can be tampered with trivially in the browser . If
you collect display name, city, country, IP-derived signals, or any online identifier , you should design the
feature as a UK GDPR / EU-style personal-data flow with data minimisation, short retention, a privacy
notice, and pseudonymisation of abuse signals. 
Assumptions used in this report: there is no fixed budget ceiling, no stated traffic limit, the first
release is one game only, modern evergreen browsers are acceptable, the site is in English, and you
want a minimal but premium UI rather than a novelty design.

Quiz engine implementation patterns
What you originally called the “duplicates” feature is more precisely overlapping or cascading auto-
submit. True duplicate handling just means preventing the same team from scoring twice. Your easy-
mode rule is something different: one entered string may intentionally satisfy more than one unseen
answer. A good internal name is overlapMode or cascadeMatches. For example, with overlap
mode enabled, typing Texas A&M may solve both Texas A&M and Texas if your rules explicitly permit
overlap. In medium and hard, the matcher should use single-resolution exact alias rules so that 
Texas A&M does not accidentally award Texas. That makes the design easy to reason about and test.
This recommendation follows directly from the need to enforce allowlist-style validation, Unicode
normalisation and deterministic downstream processing rather than fuzzy ad hoc matching. 
The browser-side implementation should listen to the input event, because it fires whenever the
user changes the entered value, but your code should not perform matching while text composition is
active. MDN exposes InputEvent.isComposing specifically for the period between 
compositionstart and compositionend, which matters if the user types with an IME. In practice:
if isComposing === true, update the visible input but defer match evaluation until composition
ends. Otherwise you risk incorrect auto-submits for East Asian text entry and some mobile keyboards. 
Normalisation should be explicit, versioned and testable. At minimum, use Unicode normalisation first,
then lowercase, collapse whitespace, harmonise punctuation variants, and run matching against a 
curated alias table. MDN shows why Unicode-equivalent strings can compare differently until
normalised, and also notes that compatibility normalisation can be useful for search-style matching.
Use this carefully: for answer checking, I would normalise to NFKC or NFC, then apply your own
punctuation rules, because you want consistency without destroying meaningful distinctions. Use 
Intl.Collator only for display sorting or locale-aware comparison helpers, not as the sole answer
validator . 
The key implementation choice is prefix-collision handling. Do not allow naïve substring checks.
Instead, build a matcher that knows the difference between: a canonical name, an alias, a prefix of
another alias, and a token-boundary-contained alias. For medium and hard, the safest rule is: submit
only on exact alias equality after normalisation. For easy overlap mode, you may emit all unseen
matches that occur at valid token boundaries, but still resolve them longest-first so that texas a&m
marks Texas A&M before it considers whether Texas should also be awarded. That prevents weirdness
like florida atlantic awarding florida unless you explicitly want that. This is where an allowlist
strategy matters: decide in data which overlaps are permitted, instead of hoping the code guesses well.

Timers should be driven from an absolute deadline, not from setInterval counts. Browsers throttle
timers and pause animation callbacks in hidden tabs, so a countdown that depends on “one second
ticked 900 times” will drift. Persist startedAt, deadlineAt, difficulty, solvedIds, 
datasetVersion and updatedAt, then recompute 
remainingMs = deadlineAt - Date.now() whenever the page restores or changes visibility. The
Page Visibility API is the correct hook for recalculating when the tab becomes visible again. This is one
of the biggest correctness wins you can make. 
For persistence and resume, localStorage is fine for a single-player quiz save because it survives
browser restarts, but remember that Web Storage is synchronous and origin-scoped. Keep the save
blob small. A good pattern is one small JSON object per active quiz mode. If you want to prevent multi-
tab confusion, use BroadcastChannel so one tab can notify the others that a timed session is already
active. With that, opening a second tab can show a non-destructive “Resume current quiz?” banner
rather than silently forking the timer . 
Accessibility should be built into the engine rather than added later . WCAG requires a visible keyboard
focus indicator , and WAI guidance stresses predictable focus movement and not losing focus during
DOM updates. Your main input can reasonably auto-focus because the page has one dominant task, but
you should not steal focus toward the grid when a team is solved. Keep focus in the input, announce
the solve in a polite live region, and reserve role="alert" / assertive messaging for errors and
urgent timer warnings like “30 seconds remaining.” Respect prefers-reduced-motion by disabling
bounce, scale and sweep animations when the user requests less movement. 
Minimal but striking UX and UI patterns
The best way to beat Sporcle aesthetically is not to add more chrome; it is to make the page feel 
cleaner, faster and more intentional. Keep the structure extremely simple: a hero-top strip with game
title, difficulty selector , progress summary and input; below that, a responsive conference grid of cards;
within each card, a compact roster table of empty slots that fill as answers are found. Make the visual
system do the work through typography, spacing, contrast and motion restraint, not through
skeuomorphic sports graphics. That is also friendlier to keyboard and mobile users. The accessibility
case for strong focus indicators and restrained motion is not at odds with premium design; it is part of
it. 
For layout, use semantic tables or lists inside conference cards because your content is
fundamentally structured roster data. On desktop, a CSS grid works well: four columns at very wide
widths, three on laptop, two on tablet, one accordion column on mobile. The accordion pattern is
especially appropriate on small screens because WAI’s APG explicitly frames it as a way to reduce the
need to scroll through multiple sections. On mobile, I would allow only one expanded conference at a
time; on desktop, keep all cards visible.

Leaderboards, privacy and security
A leaderboard changes the engineering class of the project. If the goal is just “show my score on this
device”, client-only storage is enough. If the goal is “public top scores by difficulty, live updates, abuse
resistance, and persistence across devices”, you need a server-backed validation flow. The clearest
dividing line is trust: if the browser is allowed to declare its own finish time and score without server
verification, it can be forged trivially. That makes a client-only board fine for a hobby prototype, but not
for anything competitive. 
For real-time updates, choose the simplest mechanism that fits the feature. SSE is excellent for one-way
“new leaderboard entries” feeds because the server pushes events to the browser whenever data
changes. WebSockets make sense when you want bidirectional real-time messaging or richer live state,
but they also add more operational complexity. If the leaderboard only refreshes every few seconds and
traffic is modest, polling is entirely acceptable for v1. Supabase exposes real-time patterns oriented
around Broadcast, Presence and Postgres Changes; Firebase offers real-time databases and
application-backed updates; Hasura offers GraphQL subscriptions. 
The most important privacy point is that leaderboard data is not “harmless” just because it looks game-
like. The ICO’s UK GDPR guidance defines personal data broadly enough to include names, location data
and online identifiers, and it is explicit that pseudonymised data remains personal data if it can still
be linked back with additional information. So if you collect display name, city, country, IP-derived
identifiers, or a persistent anon ID, assume you are operating a personal-data feature. The safest
baseline is: make location optional, store only what you need, publish only what the user expects to be
public, pseudonymise anti-abuse signals, set short retention for raw security logs, and document all of
it in a privacy notice.

The official documentation behind the table is straightforward: Firebase exposes a no-cost Spark plan
and pay-as-you-go Blaze pricing, Supabase offers Free and Pro ($25/month) plans, Hasura’s DDN
pricing starts at $5 per active model/month for Base, Supabase documents its real-time features
directly, and MDN differentiates clearly between WebSockets and server-sent events. 
My recommendation with your current requirements is:
Prototype / private alpha: static frontend + local saves only.
Public beta: Supabase Postgres + serverless submit validation + optional realtime “recent
finishes”.
If you want the least backend design work: Firebase is the fastest managed alternative.
If you already prefer GraphQL-heavy infra: Hasura becomes more attractive.
A secure submit flow should never trust the browser’s declared time. Instead, the server should mint a
session object at quiz start containing quizId, difficulty, datasetVersion, startedAt, 
deadlineAt, and a signature or server-side row. On submit, the server recomputes score validity,
checks that the deadline is plausible, verifies one active attempt per user/anon ID, and only then writes
the leaderboard entry. This is also where you can add anti-cheat signals such as submission velocity,
duplicate solve patterns, and impossible timings. OWASP’s anti-automation guidance is directly relevant
here: apply rate limits at more than one key, use layered controls, and do not expose overly specific
failure messages that help attackers tune their tooling.

Testing, roadmap and critical pitfalls
The testing strategy should mirror the risk areas of the product. Unit-test the normalisation and
answer-resolution engine very heavily, because that is where most hidden bugs live. Use fake timers
for timer math so the tests do not actually wait, and use browser-level end-to-end tests for keyboard
navigation, input persistence, responsive layout, and leaderboard submit flows. Playwright is a strong
fit here because it auto-waits for actionable elements and reduces flaky timing issues, while Vitest’s fake
timers make countdown logic straightforward to exercise. 
For accessibility regression testing, add an automated scan step in E2E. Playwright’s accessibility testing
guidance shows an @axe-core/playwright pattern and also notes an important limitation:
automated tooling does not catch all WCAG issues. So the right model is: automated scans for broad
regressions, plus manual passes for focus order , screen-reader announcements, reduced motion, and
mobile keyboard experience.

Prioritised roadmap
Must-have for v1: corrected 2026 dataset, difficulty modes, deterministic alias engine, timer
persistence, local resume, accessible focus/live regions, responsive conference layout, and a versioned
results model. These are the foundations of correctness and replayability. 
Nice-to-have soon after: public leaderboard by difficulty, recent finishes ticker , streak animations that
honour reduced-motion, shareable result cards, “give up / reveal remaining teams”, and an admin
dataset editor . These improve retention and maintainability without changing the core game loop. 
Later: daily challenge mode, map mode, logos mode, nickname mode, conference-only micro quizzes,
authenticated profiles, anti-cheat scoring calibration, and seasonal archive datasets such as 
fbs-2024, fbs-2025, fbs-2026. The archive idea is especially valuable because conference
membership changes make old quizzes interesting in their own right.

Final product recommendation
If you want the strongest first implementation, build the quiz itself first-class, and treat the
leaderboard as a second layer with server validation. The order of operations should be: correct
dataset -> deterministic matcher -> timed persistence -> accessible responsive UI -> leaderboard -
> admin tools. That sequence will produce a site that already feels materially better than Sporcle before
you add any “extra” features, and it will also make the Codex prompt far less likely to generate brittle or
stale code.
